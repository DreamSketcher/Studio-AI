# ТЗ: извлечение env-движка из XTTS-Studio-AI в новый проект (AI Studio / PySide6)

## 0. Роль и режим работы агента

**Модель доступа:** у тебя НЕТ доступа на запись в `DreamSketcher/XTTS-Studio-AI` и
НЕТ прав на git commit/push куда-либо. Исходный репозиторий — это read-only источник
правды: ты скачиваешь его (clone или zip) в своё локальное рабочее пространство и
работаешь только с этой локальной копией. Всё, что ты производишь — новые/изменённые
файлы в своей рабочей директории — существует только локально, пока пользователь
сам не заберёт результат (скачает выгруженную папку/архив).

Из этого следует три практических правила:

1. **Не пытайся делать `git commit`, `git push` или открывать PR** ни в исходный, ни
   в новый репозиторий — у тебя нет для этого доступа, и это не входит в задачу.
   Финальный результат — это набор файлов в твоей рабочей директории, который
   пользователь заберёт вручную.
2. **Раз нет git-истории, которая обычно служит журналом изменений, её роль здесь
   играет обязательный diff-отчёт** (раздел 2.4) — без него исправления в env_core
   невозможно будет ни проверить, ни откатить позже, потому что commit-лога не будет.
3. Работа физически идёт в один проход по твоему локальному пространству: сначала
   правишь скачанную копию (логический "Этап A"), затем прямо из неё собираешь
   файловое дерево нового проекта (логический "Этап B"). Это не два разных
   репозитория с git-операциями между ними — это копия → правка → раскладка по новой
   структуре папок, всё в одной рабочей директории.

Не смешивай логические этапы местами (не раскладывай файлы по новому проекту, пока
правки из Этапа A не прошли приёмку, раздел 5) — но физически это одна сессия работы
с файлами, без переключения между репозиториями.

Если по ходу работы возникает необходимость отступить от этого ТЗ (например, найден
ещё один скрытый импорт GUI, не описанный здесь) — **останавливайся и сообщай об этом
текстом**, не принимай архитектурное решение самостоятельно молча. Тихие компромиссы
здесь дороже, чем задержка на подтверждение.

## 2. Логический этап A — исправление связности на локальной копии-источнике

Скачай `DreamSketcher/XTTS-Studio-AI` (ветка `main`) в отдельную локальную папку,
например `./_source_xtts_studio/`. Эта папка — временный рабочий материал, она не
уходит никуда как отдельный репозиторий и не коммитится; она существует только
затем, чтобы из неё вырезать уже исправленные файлы на шаге 3.

### 2.1. Устранить нарушения #1–#3 (install-lock state живёт не в том слое)

**Диагноз:** функции `_acquire_install_lock`, `_release_install_lock`,
`_get_current_install_type`, `_set_install_cancelled`, `_can_clear_diagnostics_cache`
по смыслу — это состояние окружения (env), а не состояние интерфейса (gui). Они
оказались в `engine/gui/env_settings.py` исторически, а не по архитектурной причине.

**Действие:**
1. Создать новый модуль `engine/env_core/install_state.py`.
2. Перенести туда определения всех пяти функций дословно, без изменения логики
   (это чистый move, не rewrite — сверь построчно после переноса, что поведение
   идентично оригиналу).
3. В `engine/gui/env_settings.py` заменить прежние определения на
   `from engine.env_core.install_state import (...)` — GUI теперь читает состояние
   из env_core, а не наоборот. Публичные имена, вызываемые из остального GUI-кода,
   не должны измениться (никаких переименований, чтобы не тянуть правки по всему
   `engine/gui/`).
4. В `engine/env_core/rvc_setup.py` и `engine/env_core/torch_setup.py` заменить
   `from engine.gui.env_settings import ...` на
   `from engine.env_core.install_state import ...` — импорт становится
   внутрипакетным, try/except-заглушки удалить (они больше не нужны, модуль всегда
   доступен внутри своего же пакета).

**Приёмка шага:** `lint-imports --config .importlinter` — контракты
`env_core_independence` и `ai_modules_independence` переходят в `KEPT` без записи
в `ignore_imports`. Убрать соответствующие две строки из `ignore_imports` в
`.importlinter` — если контракт всё ещё ломается, шаг не закрыт.

### 2.2. Устранить нарушение #4 (`gpt_client.py` → `i18n`)

**Действие:** заменить жёсткий импорт на безопасный с фолбэком:

```python
try:
    from i18n import t as _t
except ImportError:
    def _t(key: str) -> str:
        return key
```

Не переписывать вызовы `_t(...)` по всему файлу — фолбэк-функция с той же сигнатурой
делает это ненужным.

**Приёмка шага:** `python -c "import engine.gpt_client"` из окружения без `i18n.py`
в PYTHONPATH завершается без ImportError.

### 2.3. Полный прогон проверки после 2.1–2.2

```bash
pip install import-linter packaging
lint-imports --config .importlinter
```

Ожидаемый результат: все три контракта — `KEPT`, `Contracts: 3 kept, 0 broken`.
Это жёсткий критерий приёмки Этапа A. Не переходить к Этапу B, пока это не так.

### 2.4. Регрессионный прогон существующих тестов

```bash
pytest test/ -v
```

Все существующие тесты (`test_updater.py`, `test_normalizer.py`, `test_chunker.py`,
`test_smart_pauses.py` и другие в `test/`) должны проходить без изменений в их
собственном коде. Если что-то падает после 2.1–2.2 — это сигнал, что перенос
`install_state.py` что-то сломал по факту (не только по графу импортов), и это
нужно исправить, а не подавить пропуском теста.

### 2.5. Обязательный diff-отчёт (замена git-истории)

Поскольку правки не коммитятся никуда, единственный способ впоследствии проверить,
что именно было изменено на этом шаге — файл-отчёт, который ты обязан создать перед
переходом к разделу 3:

```
_source_xtts_studio_CHANGES.diff
```

Сгенерировать его так (сравнение с чистой копией, снятой ДО правок 2.1–2.2):

```bash
diff -u <оригинальный_файл_до_правок> <файл_после_правок> >> _source_xtts_studio_CHANGES.diff
```

Отчёт должен содержать unified diff по каждому из четырёх изменённых файлов
(`engine/gui/env_settings.py`, `engine/env_core/rvc_setup.py`,
`engine/env_core/torch_setup.py`, `engine/gpt_client.py`) и по одному новому файлу
(`engine/env_core/install_state.py` — для нового файла привести его содержимое целиком,
diff тут не нужен). В начале отчёта — короткая шапка: дата, что и зачем менялось,
результат прогона `lint-imports` (raздел 2.3) и `pytest` (раздел 2.4) до и после.

Этот файл — часть финальной поставки (раздел 6), он должен лежать рядом с новым
проектом, а не потеряться в рабочей папке `_source_xtts_studio/`.

## 3. Логический этап B — раскладка в файловую структуру нового проекта

Создай в рабочем пространстве отдельную папку для нового проекта, например
`./ai_studio_core/` (имя уточни, если пользователь дал конкретное). Это НЕ git-клон
чего-либо — это просто новая директория, куда копируются уже исправленные файлы
из `_source_xtts_studio/`. Никакого `git init`/`git remote` делать не нужно —
подготовку репозитория (создание на GitHub, первый коммит) пользователь делает сам
после того, как заберёт готовую папку.

### 3.2. Правила переноса (что можно/нельзя менять при переносе)

- **Нельзя** менять логику при переносе файла. Перенос = `git mv` эквивалент +
  правка путей импорта, если поменялось имя корневого пакета. Если родился соблазн
  "заодно улучшить" — не делать в рамках этого ТЗ, заводить отдельную задачу.
- **Можно и нужно** поменять корневой пакет в импортах, если новый проект называет
  его иначе (например `engine` → `ai_studio_core`). Делать это одним поиском-заменой
  по всему перенесённому дереву, не точечно.
- **Нельзя** оставлять закомментированный старый код "на всякий случай" — либо
  переносится рабочая версия, либо не переносится вообще.
- `i18n`-фолбэк в `gpt_client.py` (см. 2.2) остаётся как есть — новый проект может
  либо не иметь `i18n.py` вовсе на этом этапе, либо завести свою реализацию с той же
  сигнатурой `t(key: str) -> str` позже.

### 3.3. Настройка графа импортов в новом проекте

1. Скопировать `.importlinter` в корень нового репозитория.
2. Обновить `root_package` и имена слоёв под новое имя корневого пакета.
3. Убрать секцию `ignore_imports` полностью — в новом репозитории с первого коммита
   не должно быть ни одного зафиксированного технического долга по нарушению слоёв.
4. Добавить в `models/` (или аналогичную директорию) хотя бы один тестовый манифест
   в формате, читаемом `check_plugin_deps.py` (см. ранее переданный инструмент), даже
   если реальных моделей-плагинов ещё нет — это фиксирует формат раньше первой
   реальной модели, а не после.

### 3.4. CI нового проекта

Перенести `.github/workflows/architecture_gate.yml` (уже подготовлен) без
изменений, кроме путей, если структура каталогов отличается. Добавить отдельный
job матрицы `windows-latest / macos-latest / ubuntu-latest`, который:
- ставит зависимости перенесённого `engine`-слоя (без GUI-зависимостей, т.к. они
  не перенесены);
- запускает существующие pytest-тесты (`test_updater.py`, `test_normalizer.py`,
  `test_chunker.py`, `test_smart_pauses.py`), скопированные вместе с кодом;
- запускает `check_architecture.py` и `check_plugin_deps.py`.

## 4. Финальная поставка (что должно остаться в рабочей папке)

По завершении работы в рабочем пространстве агента должны лежать три независимых
артефакта — пользователь заберёт их вручную, поэтому важно не смешивать их в одну
кучу:

```
_source_xtts_studio/              — рабочая копия исходника с правками 2.1–2.2
                                     (можно удалить после того, как ai_studio_core/
                                     проверен, но НЕ удалять раньше готовности пункта 3)
_source_xtts_studio_CHANGES.diff  — обязательный diff-отчёт (раздел 2.5)
ai_studio_core/                   — готовая файловая структура нового проекта,
                                     это и есть основной результат задачи
```

Не удаляй `_source_xtts_studio/` и `_source_xtts_studio_CHANGES.diff` самостоятельно
после копирования — пользователь должен иметь возможность свериться с ними, если
что-то в `ai_studio_core/` вызовет вопросы.

## 5. Критерии приёмки всей задачи

Задача считается выполненной, когда одновременно верно (проверяется локально, в
рабочей папке агента — ничего из этого не требует git commit/push):

1. В `_source_xtts_studio/` (правленая копия) `.importlinter` проходит с
   `Contracts: 3 kept, 0 broken` и без единой строки в `ignore_imports`.
2. Существующие тесты (`pytest test/` внутри `_source_xtts_studio/`) проходят без
   изменений в тестовом коде.
3. В `ai_studio_core/` перенесённый `engine`-слой импортируется и инициализируется
   (`python -c "import <new_package>.env_core"`) на чистом окружении **без**
   установленных `tkinter`/`customtkinter`/`PySide6` — то есть доказано, что перенос
   действительно headless, а не "почти headless".
4. `lint-imports` в `ai_studio_core/` проходит с 0 нарушений и без `ignore_imports`.
5. `_source_xtts_studio_CHANGES.diff` присутствует, читаем и покрывает все файлы
   из раздела 2.5.
6. Готов ли CI-workflow нового проекта к прогону — оценить статически (сам агент
   не имеет доступа к GitHub Actions для реального прогона на трёх ОС, это сделает
   пользователь после того, как заберёт `ai_studio_core/` и создаст под него
   репозиторий) — но конфиг должен быть на месте и синтаксически корректен.

## 6. Что явно вне рамок этой задачи (чтобы агент не расширял скоуп)

- Написание PySide6 GUI-обвязки поверх перенесённого env_core — отдельная задача,
  идёт по устройству, описанному в уставе проекта (Этап 2 и далее), не здесь.
- Добавление новых AI-моделей/плагинов — отдельная задача, наступает после Этапа 3
  устава (plugin-архитектура на dummy-модели), не раньше.
- Рефакторинг логики диагностики/апдейтера сверх устранения пунктов 2.1–2.2 — не
  делать "заодно", даже если по ходу работы найдутся другие улучшаемые места.
